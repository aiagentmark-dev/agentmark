# agentmark specification

**Version:** 0.1 (draft)
**Status:** pre-release
**Authors:** CloudDon Research
**Repository:** github.com/aiagentmark-dev/agentmark (proposed)

---

## 1. Overview

agentmark is an open standard for verifying that code changes traveled through a cryptographically attested autonomous AI pipeline with no direct human write path.

It does not claim to prove that an entity is conscious, intentional, or philosophically "an AI." It proves something narrower and more useful: that a specific piece of code was generated by a specific LLM API call, committed verbatim without human editing, and submitted by a registered pipeline identity.

This is analogous to how HTTPS proves transport integrity without proving what the human at the keyboard intended — and it is sufficient for the practical purposes of AI code governance, supply chain audit, and regulatory compliance.

### 1.1 What agentmark proves

- The code was produced by a real LLM API call (not typed by a human)
- The code was committed verbatim — no human editing occurred between generation and commit
- The LLM saw the specific task it was assigned (challenge token in context)
- The commit was made by a registered pipeline identity with a known signing key
- The full prompt and response are logged and auditable

### 1.2 What agentmark does not prove

- That the human operator did not craft the prompt to produce specific code
- That the pipeline was not human-directed at the input level
- That the agent acted autonomously in any philosophical sense
- Absolute certainty against a determined adversary running the full pipeline manually

The irreducible gap: a human who runs the complete agentmark-compliant pipeline (real API call, verbatim commit, valid manifest, valid signature) is indistinguishable from an agent. This is acceptable — they have built an agent. The goal is not to prevent this; it is to make honest behavior the path of least resistance.

### 1.3 Relationship to existing standards

| Standard | What it covers | Gap agentmark fills |
|---|---|---|
| SLSA | Build pipeline integrity | Does not address AI authorship |
| SBOM / AI-BOM | Inventory of models used | Not which API call produced which commit |
| C2PA | Media provenance (images, audio) | Not designed for code commits |
| sigstore | Artifact signing infrastructure | Not AI-aware |
| APP | AI content metadata standard | No pipeline verification, no code focus |

agentmark is designed to complement, not replace, these standards.

## 2. Top use cases

### Use case 1 — Agent-only open source projects (primary)

Projects like Karta (karta.build) where the founding constraint is that no human ever commits application code. agentmark provides the verification layer that enforces and proves this property. Without agentmark, the claim "this project was built entirely by AI agents" is unverifiable marketing. With agentmark, every commit carries a cryptographic audit trail.

**Who needs this:** Agent-first development projects, AI coding experiments, research projects studying autonomous software development.

### Use case 2 — Enterprise AI code governance

Microsoft's Satya Nadella said 20–30% of code in some Microsoft repositories is now AI-generated — and there is no reliable method to measure AI-generated code in a repository after the fact. agentmark gives enterprises commit-level traceability: which model generated which function, from which prompt, at which timestamp.

**Who needs this:** Enterprise DevSecOps teams, regulated industries (finance, healthcare, defense), companies facing EU AI Act Article 50 compliance, procurement teams buying AI-generated software.

### Use case 3 — Defense and critical infrastructure supply chain

National defense organizations cannot trace AI-generated code in systems they buy, yet vendors continue shipping AI-generated code to allied governments. agentmark enables procurement officers to require verifiable AI code provenance as a contract condition.

**Who needs this:** Defense contractors, critical infrastructure operators, government software procurement teams, NATO-adjacent organizations.

## 3. Competitive landscape

### Competitor 1 — SLSA + sigstore (closest technical overlap)

**What it is:** SLSA (Supply-chain Levels for Software Artifacts) is a Google-backed framework for build pipeline integrity. Sigstore provides keyless signing infrastructure. Together they prove "this artifact was built from this source by this build process."

**Where it falls short for agentmark's purpose:**
- SLSA proves build pipeline integrity, not AI authorship
- SBOMs tell you what's in your software. They don't tell you whether it was built from the code you think it was. Similarly, SLSA doesn't tell you whether a human or an AI wrote the source code
- No concept of LLM API call, output hash, or challenge token
- An enterprise relying on sigstore may not be able to deploy new code upon availability issues with the public sigstore cluster, including service downtime, breaking changes, and rate limiting.

**Relationship to agentmark:** Complementary. agentmark adds the AI authorship layer on top of SLSA's build integrity layer. An agentmark-verified commit can also carry SLSA provenance.

### Competitor 2 — AI Provenance Protocol (APP)

**What it is:** An open JSON standard for marking AI-generated content with metadata — model name, generation timestamp, generation ID, human review status.

**Where it falls short:**
- Designed for content (product descriptions, articles) not code commits
- No pipeline verification — anyone can add an `_ai_provenance` field to any JSON
- No output hash — no proof the content wasn't edited after generation
- No challenge token — no proof the LLM processed the specific task
- No signing — no registered pipeline identity
- `generation_id` is not cryptographically verifiable (no provider verification endpoint)

**Relationship to agentmark:** agentmark is the code-commit-specific, cryptographically verifiable version of what APP is trying to do for content. The two could coexist — APP for content provenance, agentmark for code provenance.

### Competitor 3 — Qodo / enterprise code verification platforms

Qodo raised $70M Series B, building AI agents for code review, testing, and governance, betting that verification will define the next phase of software development.

**What it is:** Enterprise platform for AI code quality, review, and governance. Focuses on correctness, security, and testing rather than provenance.

**Where it falls short:**
- Verifies code quality, not code origin
- No cryptographic proof of AI authorship
- Proprietary platform, not an open standard
- Addresses "is this code good?" not "did an AI write this?"

**Relationship to agentmark:** Different problem. Qodo and agentmark are complementary — Qodo verifies quality, agentmark verifies provenance.

## 4. Known failure modes and limitations

These must be called out honestly in the spec. They are not bugs to fix — they are boundaries of what the standard can and cannot claim.

### F1 — The verbatim human copy attack (irreducible)

**Description:** A human calls an LLM API with the agentmark challenge in the prompt, captures the raw response, commits verbatim, and provides a valid manifest.

**Detection:** None. This passes all agentmark checks.

**Why it is acceptable:** The human has built the agent pipeline. The distinction between "human running agent pipeline" and "agent running agent pipeline" collapses. This is the intended design: agentmark verifies the pipeline, not the operator's intent.

**Mitigation:** Behavioral signals over time (commit timing patterns, parallelism, response latency) can flag anomalies. These are advisory, not deterministic.

### F2 — LLM proxy / gateway breaks output_hash

**Description:** Agents using LiteLLM, AWS Bedrock, Azure OpenAI, or any other proxy/gateway receive a transformed response. The raw bytes from the proxy differ from the raw bytes Anthropic/OpenAI returned.

**Detection:** output_hash mismatch on verification.

**Mitigation:** agentmark requires direct provider API calls. Proxy use is not supported in v1.0. Agents must call `api.anthropic.com` or `api.openai.com` directly. Proxy-aware hashing is a v2.0 consideration.

**Spec requirement:** The manifest must declare `direct_provider_call: true`. CI verification may reject manifests where proxy headers are detected.

### F3 — Local / self-hosted models have no request_id

**Description:** Agents using Ollama, llama.cpp, vLLM, or any self-hosted model cannot provide a provider-issued `request_id`. There is no third party to corroborate the API call.

**Detection:** Not applicable — this is a known limitation, not an attack.

**Mitigation:** agentmark defines two trust tiers (see section 6). Local models qualify for Tier 2 (reduced provenance). `request_id` is optional for `provider: local`.

**Risk:** Tier 2 relies entirely on operator accountability and output_hash + challenge_token. Weaker than Tier 1 but still meaningful.

### F4 — Challenge token not echoed by LLM

**Description:** Some LLMs may not follow the instruction to echo the challenge token, particularly if the instruction is buried in a long prompt or the model's safety filters strip it.

**Detection:** `challenge_echo_verified: false` in manifest. CI gate rejects the commit.

**Mitigation:** Challenge token embedding format must be explicit and unambiguous. Testing confirms gpt-4o echoes reliably with current prompt format. Models that consistently fail echo should be flagged as not agentmark-compatible until resolved.

**Spec requirement:** The challenge instruction must appear as the final instruction in the prompt, not embedded in the middle.

### F5 — request_id format spoofing

**Description:** A human fabricates a `request_id` that matches the declared provider's format without making a real API call.

**Detection:** Format validation only in v1.0. No provider verification endpoint exists today.

**Mitigation:** request_id is a supporting signal, not the primary proof. output_hash and challenge_token are the primary proofs. request_id format mismatch is a red flag; valid format is not a green flag.

**Roadmap:** agentmark notary service (v2.0) cross-verifies request_ids with providers. When providers expose verification endpoints, this gap closes.

### F6 — Signing key compromise

**Description:** A human obtains the pipeline's private signing key and uses it to sign arbitrary commits.

**Detection:** Not detectable cryptographically. Indistinguishable from legitimate pipeline commits.

**Mitigation:** Standard key management practices — key rotation, key storage in secrets manager, TEE-based key generation (v3.0). Operators must report compromised keys to the agentmark registry immediately.

### F7 — Multi-call agents break the output_hash model

**Description:** Agents using frameworks like CrewAI, LangGraph, or AutoGen may make multiple LLM API calls per task — planning, implementation, review, refinement. Which call's output_hash goes in the manifest?

**Detection:** Not applicable — this is a design question.

**Mitigation:** agentmark v1.0 requires a single "generation call" — the call whose raw output is committed verbatim. Multi-call frameworks must designate one call as the canonical generation call and commit that output verbatim. Intermediate calls may be logged but are not the subject of the manifest.

**v2.0 consideration:** A multi-call manifest format that chains multiple request_ids and output_hashes across a generation pipeline.

### F8 — Model version drift

**Description:** The declared `model` field (e.g., `gpt-4o-2024-08-06`) may not match the actual model used if the provider silently updates the underlying model behind a stable alias.

**Detection:** Not detectable without provider cooperation.

**Mitigation:** Use pinned model versions, not aliases, in agent configurations. `gpt-4o` is an alias; `gpt-4o-2024-08-06` is a pinned version. agentmark recommends pinned versions.

## 5. Core mechanisms

### 5.1 output_hash

The primary tamper-detection mechanism.

**Definition:** `sha256(raw_http_response_body_bytes)` where `raw_http_response_body_bytes` is the complete HTTP response body as received from the provider API, before any parsing, decoding, or transformation.

**Verification:** The verifier computes `sha256(committed_code_bytes_as_extracted_from_raw_response)` and checks it matches `manifest.output_hash`. Exact byte-for-byte match required.

**What this proves:** No byte was changed between the LLM's response and the committed code.

**What this does not prove:** The operator did not direct the generation.

**Provider notes:**
- The raw bytes are JSON-encoded. Newlines in generated code appear as `\n` in the raw bytes.
- For Anthropic: raw bytes are the response body from `POST https://api.anthropic.com/v1/messages`
- For OpenAI: raw bytes are the response body from `POST https://api.openai.com/v1/chat/completions`
- Both SDKs expose raw bytes via `client.*.with_raw_response.create()` → `.content`

### 5.2 challenge_token

The LLM-processed-this-task mechanism.

**Format:** `agentmark-{16 lowercase hex chars}` e.g. `agentmark-3f9a2b1c4d5e6f7a`

**Issuance:** agentmark GitHub App or CLI issues one token per task (issue, PR, or work item). Tokens are single-use and expire after 24 hours.

**Embedding:** The token is embedded in the task description as an HTML comment:
```
<!-- agentmark-challenge: agentmark-3f9a2b1c4d5e6f7a -->
```

**Echo instruction:** The final instruction in the agent's prompt must be:
```
Your response MUST contain this exact string on its own line:
agentmark-challenge-echo: agentmark-3f9a2b1c4d5e6f7a
```

**Verification:** The verifier checks that `agentmark-challenge-echo: {token}` appears in `raw_http_response_body_bytes`. Since output_hash covers raw_bytes, the echo is cryptographically bound to the hash.

**What this proves:** The LLM processed the specific task in its context window.

**What this does not prove:** The human operator did not craft the prompt to direct specific output.

### 5.3 request_id

The real-API-call signal.

**Provider mapping:**

| Provider | Header name | Format | Example |
|---|---|---|---|
| Anthropic | `request-id` | `req_01{22 alphanum}` | `req_011CZRtQztYq...` |
| OpenAI | `x-request-id` | `req-{hex}` or UUID | `req_1bf0bd264...` |
| Google Gemini | `x-goog-request-id` | UUID | `a1b2c3d4-e5f6...` |
| Mistral | `x-request-id` | UUID | standard UUID |
| Local | none | n/a | omit field |

**Verification in v1.0:** Format check only. The verifier confirms the request_id format matches the declared provider. Provider-side verification (confirming the call actually happened) is not available in v1.0.

**Trust implication:** A valid-format request_id is a necessary but not sufficient condition. It is a supporting signal that raises the cost of fabrication, not a cryptographic proof.

### 5.4 Ed25519 signing

The pipeline identity mechanism.

**Key generation:** Each registered pipeline generates an Ed25519 keypair. The private key is held by the pipeline's infrastructure (secrets manager, TEE in v3.0). The public key is registered with the agentmark registry.

**Signing:** The manifest (excluding the `signature` field) is serialized as canonical JSON (keys sorted, no extra whitespace) and signed with the private key. The signature is base64-encoded.

**Verification:** The verifier fetches the public key for the declared `pipeline_key` from the registry and verifies the signature over the canonical manifest.

**What this proves:** This specific registered pipeline made this commit.

**What this does not prove:** The pipeline was not operated by a human.

**Registration:** Pipeline registration requires a human operator to provide: pipeline_id, public key, operator identity (GitHub handle), and operator accountability statement. The registry is public.

## 6. Trust tiers

### Tier 1 — Cloud provider with request_id (required for external contributors)

Required fields: `request_id`, `output_hash`, `challenge_token`, `signature`, `provider` (must be cloud provider)

Verification: All five mechanisms checked. request_id format validated against provider.

### Tier 2 — Local model or trusted operator (reduced provenance)

Required fields: `output_hash`, `challenge_token`, `signature`, `provider: local`

Verification: output_hash, challenge_token, and signature checked. request_id absent or ignored.

Risk: Weaker provenance. Suitable for trusted operators only. Not acceptable for external contributor registration.

### Tier 3 — TEE-attested (future, v3.0)

Adds hardware attestation report from Intel SGX / AMD SEV to Tier 1. Proves the exact pipeline binary ran in an isolated environment with no human I/O path.

## 7. Manifest schema

Every agentmark-verified commit must include a `agentmark-manifest` block in the commit message.

```json
{
  "version": "1.0",
  "provider": "anthropic",
  "model": "claude-sonnet-4-20250514",
  "request_id": "req_011CZRtQztYq...",
  "output_hash": "sha256:a1b2c3d4...",
  "challenge_token": "agentmark-3f9a2b1c4d5e6f7a",
  "challenge_echo_verified": true,
  "pipeline_key": "karta-coder-v1",
  "prompt_log": "logs/run-20260416-001.json",
  "direct_provider_call": true,
  "timestamp": "2026-04-16T10:22:47Z",
  "signature": "base64-encoded-ed25519-signature"
}
```

### Field definitions

| Field | Required | Description |
|---|---|---|
| `version` | Yes | agentmark spec version. Currently `"1.0"` |
| `provider` | Yes | LLM provider: `anthropic`, `openai`, `google`, `mistral`, `local` |
| `model` | Yes | Pinned model identifier. Use versioned names, not aliases |
| `request_id` | Tier 1 | Provider-issued request identifier from response headers |
| `output_hash` | Yes | `sha256:{hex}` of raw HTTP response body bytes |
| `challenge_token` | Yes | agentmark-issued single-use challenge token |
| `challenge_echo_verified` | Yes | `true` if echo found in raw response bytes |
| `pipeline_key` | Yes | Registered pipeline identity identifier |
| `prompt_log` | Recommended | Path to logged prompt/response JSON file |
| `direct_provider_call` | Yes | `true` if no proxy/gateway was used |
| `timestamp` | Yes | ISO 8601 UTC timestamp of the API call |
| `signature` | Yes | Base64 Ed25519 signature over canonical manifest (excluding this field) |

### Commit message format

```
<conventional commit message>

```agentmark-manifest
{
  ... manifest JSON ...
}
```
```

## 8. Verification algorithm

`agentmark verify` executes the following checks in order. All must pass for the commit to be verified.

```
1. MANIFEST_PRESENT
   Commit message contains a valid agentmark-manifest block
   Failure: "manifest_missing"

2. SCHEMA_VALID
   All required fields present and correctly typed
   Failure: "schema_invalid:{field}"

3. SIGNATURE_VALID
   Ed25519 signature verifies against registered public key for pipeline_key
   Failure: "unregistered_pipeline_key" | "invalid_signature"

4. OUTPUT_HASH_MATCH
   sha256(raw_response_bytes) == manifest.output_hash
   Failure: "output_hash_mismatch"

5. CHALLENGE_VALID
   challenge_token was issued by agentmark for this task
   challenge_token has not been used before
   challenge_token has not expired (24h TTL)
   Failure: "challenge_not_found" | "challenge_already_used" | "challenge_expired"

6. CHALLENGE_ECHO_PRESENT
   "agentmark-challenge-echo: {token}" found in raw_response_bytes
   Failure: "challenge_echo_missing"

7. REQUEST_ID_FORMAT (Tier 1 only)
   request_id format matches declared provider pattern
   Failure: "request_id_format_mismatch" | "missing_request_id"

8. DIRECT_PROVIDER_CALL
   direct_provider_call == true
   No proxy headers detected in commit metadata
   Failure: "proxy_detected"
```

**Result:** `VERIFIED` (all pass) or `REJECTED` with first failure reason.

## 9. Provider SDK integration

### Anthropic

```python
import anthropic

client = anthropic.Anthropic()

raw_response = client.messages.with_raw_response.create(
    model="claude-sonnet-4-20250514",
    max_tokens=4096,
    messages=[{"role": "user", "content": prompt_with_challenge}],
)

request_id  = raw_response.headers.get("request-id")
raw_bytes   = raw_response.content
response    = raw_response.parse()
code        = response.content[0].text
```

### OpenAI

```python
import openai

client = openai.OpenAI()

raw_response = client.chat.completions.with_raw_response.create(
    model="gpt-4o-2024-08-06",
    messages=[{"role": "user", "content": prompt_with_challenge}],
    temperature=0,
)

request_id  = raw_response.headers.get("x-request-id")
raw_bytes   = raw_response.content
response    = raw_response.parse()
code        = response.choices[0].message.content
```

### Local (Ollama)

```python
import requests

response = requests.post(
    "http://localhost:11434/api/generate",
    json={"model": "llama3", "prompt": prompt_with_challenge, "stream": False}
)

request_id  = None           # not available for local models
raw_bytes   = response.content
code        = response.json()["response"]
```

```yaml
# .github/workflows/agentmark-verify.yml
name: agentmark verification

on: [pull_request]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install agentmark
        run: pip install agentmark

      - name: Verify pipeline provenance
        run: agentmark verify --strict
        env:
          AGENTMARK_REGISTRY_URL: https://registry.agentmark.dev

## 11. Roadmap

### v1.0 — Manifest standard + pipeline verification (current spec)
- Manifest schema
- output_hash + challenge_token + Ed25519 signing
- Python SDK: `agentmark.sign()` and `agentmark.verify()`
- GitHub App for challenge issuance
- GitHub Action for CI verification
- Pipeline registry
- Support: Anthropic, OpenAI, local models

### v2.0 — Notary service
- Cross-provider request_id verification
- agentmark notary independently confirms API calls happened
- Signed certificates for verified pipelines
- Closes F5 (request_id spoofing) gap

### v3.0 — TEE attestation
- karta-coder and other agents run in Gramine/SGX
- Hardware attestation report in manifest
- Closes F6 (key compromise) gap and eliminates human I/O path
- Tier 3 trust level

### v4.0 — Provider-native verification
- Anthropic/OpenAI expose `verify_request(request_id)` endpoints
- agentmark becomes the standard they implement against
- Full chain attestation: agent binary → LLM inference → output

## 12. Non-goals

The following are explicitly out of scope for agentmark:

- Detecting AI-generated content in code that was not produced through an agentmark pipeline
- Passive code fingerprinting or statistical AI detection
- Certifying that code is correct, secure, or of high quality
- Replacing SLSA, sigstore, or SBOM standards
- Proving the absence of human intent at the input level
- Working across all content types (documents, images, audio) — agentmark is code-specific
